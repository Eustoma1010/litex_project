# Báo cáo Phân tích Framework SoC LiteX

Báo cáo này phân tích các luồng hoạt động chính của LiteX theo các yêu cầu của bạn, có đính kèm các đoạn code ví dụ minh họa và liên kết trực tiếp tới các tệp mã nguồn trong dự án.

> [!NOTE]
> Mình đã thay đổi toàn bộ mã nguồn `firmware.c` và các tệp Makefile để hệ thống có thể kết nối với mã C tự sinh từ LiteX.

---

## 1. Thiết kế IP — Gọi các IP core mở sẵn qua lớp Python

LiteX cung cấp sẵn một hệ sinh thái các khối IP mở (như RAM, Ethernet, PCIe, SPI,...) được viết hoàn toàn dưới dạng mã Python. Khác với quy trình thiết kế Verilog truyền thống (phải viết hàng ngàn dòng mã trạng thái và khai báo tín hiệu), LiteX cho phép khởi tạo chúng cực kỳ ngắn gọn.

**Các file chứa IP tiêu biểu trong thư mục:**
- **Lõi DRAM:** [litedram/litedram/modules.py](file:///data/worktime/litex/litedram/litedram/modules.py#L448) (Chứa danh sách cấu hình của vô số chip SDRAM thật trên thị trường).
- **Lõi Mạng:** Các mô đun mạng nội bộ nằm tại thư mục [liteeth/](file:///data/worktime/litex/liteeth).

**Đoạn code minh họa sự nhanh chóng của Python:**
```python
from litedram.modules import IS42S16160
from litedram.phy import GENSDRPHY

# 1. Khởi tạo Physical Layer (PHY) cho bộ nhớ SDRAM
# Trong Verilog, việc này đòi hỏi hàng trăm dòng code để căn chỉnh độ trễ và xung nhịp
sdrphy = GENSDRPHY(pads)

# 2. Khởi tạo Memory Controller hỗ trợ chip IS42S16160 chỉ bằng 1 câu lệnh!
# LiteX sẽ tự động biên dịch cấu hình này thành các tín hiệu RTL tương ứng.
sdram = LiteDRAM(phy=sdrphy, module=IS42S16160(sys_clk_freq))
```

## 2. Tích hợp (Wiring) — LiteX tự sinh RTL kết nối & phân bổ địa chỉ

Sau khi cấu hình trên Python, hệ thống LiteX tự động nối dây giữa các linh kiện (CPU, RAM, UART) qua hệ thống Bus như Wishbone, tự phân bổ vùng nhớ (Address Mapping), và xuất ra mã Verilog. 

Kết quả tự sinh nằm tại file: [build/sim/gateware/sim.v](file:///data/worktime/litex/build/sim/gateware/sim.v#L23-L32)

**Phân tích đoạn code RTL được tạo ra trong `sim.v`:**
```verilog
// -----------------------------------------------------------------------------
// Auto-Generated by LiteX (Mã nguồn tự sinh, không nên tự chỉnh sửa thủ công)
// -----------------------------------------------------------------------------

// Module sim đóng vai trò là Top-Level chứa toàn bộ hệ thống
module sim (
    input  wire    [7:0] serial_sink_data,
    output wire          serial_sink_ready,
    // ... Khai báo tín hiệu tự sinh
);

/*
Chú thích cấu trúc phân cấp (Hierarchy) được tự động nối bằng phần mềm:
SimSoC
├── bus (SoCBusHandler - Bộ định tuyến dữ liệu Wishbone)
├── csr (SoCCSRHandler - Tự động giải quyết xung đột địa chỉ thanh ghi)
├── cpu (VexRiscv - CPU Softcore)
├── uart (UART - Khối điều khiển nối tiếp)
└── timer0 (Timer - Bộ định thời hệ thống)
*/
```

## 3. Đồng bộ HW/SW — C Header (CSR) khớp 100% với RTL

Để phần mềm có thể tương tác với phần cứng mà không phải quản lý địa chỉ thủ công, LiteX sinh ra file [csr.h](file:///data/worktime/litex/build/sim/software/include/generated/csr.h) khớp chính xác với hệ thống. 

**Những thay đổi mình đã cấu hình   lại trong project của bạn:**
1. **Sửa đổi Makefile:** Cập nhật file [Makefile.fw](file:///data/worktime/litex/Makefile.fw#L12) thêm tùy chọn `-I build/sim/software/include` vào `CFLAGS` để có thể nhận diện thư mục header tự động sinh.
2. **Tạo stub header:** Bổ sung file [system.h](file:///data/worktime/litex/build/sim/software/include/system.h) để bypass các thư viện nội bộ khác của LiteX.
3. **Cập nhật lại `firmware.c`:** [firmware.c](file:///data/worktime/litex/firmware.c) giờ đây không còn các giá trị tĩnh `#define UART_BASE 0xF0006000` dễ sinh lỗi nữa, thay vào đó:

**Code Firmware Mới (tự ánh xạ địa chỉ CSR):**
```c
// Bỏ qua các #define tĩnh thủ công. Thêm các hàm đọc ghi thanh ghi cấp thấp:
static inline uint32_t csr_read_simple(uint32_t addr) {
    return *(volatile uint32_t *)addr;
}
static inline void csr_write_simple(uint32_t value, uint32_t addr) {
    *(volatile uint32_t *)addr = value;
}

// Báo cho thư viện biết ta đã tự định nghĩa bộ truy cập
#define CSR_ACCESSORS_DEFINED
// Nạp trực tiếp tệp mapping được sinh ra từ quá trình build phần cứng
#include <generated/csr.h>

void uart_putc(char c) {
    // Gọi hàm uart_tx_ready(), bên trong hàm này sẽ sử dụng uart_txfull_read() 
    // được LiteX tự sinh trong csr.h (đã ánh xạ đúng địa chỉ thanh ghi UART hiện tại)
    while (!uart_tx_ready()) {}
    uart_rxtx_write((uint32_t)c);
}
```

## 4. Biên dịch (Build) — LiteX chạy toolchains qua CLI

Không cần phải bật phần mềm đồ họa nặng nề, LiteX tương tác trực tiếp với các trình tổng hợp của Yosys, Xilinx (Vivado), Intel (Quartus) hoàn toàn dưới nền Command Line để sinh ra Bitstream nạp xuống FPGA.

**Câu lệnh cấu hình biên dịch Vivado qua CLI:**
```bash
# Lệnh gọi Board mẫu 'Digilent Arty', yêu cầu biên dịch (--build) bằng công cụ (--toolchain) là Vivado.
python3 -m litex_boards.targets.digilent_arty --build --toolchain vivado
```
*Giải thích: Khi gọi lệnh này, hệ thống Python của LiteX sẽ sinh ra RTL, tự động viết một tệp Script TCL (.tcl), sau đó gọi Vivado thực thi ẩn ở chế độ `batch mode` (không có giao diện) để tổng hợp ra file nhị phân `.bit`.*

## 5. Nạp & Chạy — Bitstream + Nạp Firmware qua litex_term

LiteX tích hợp sẵn công cụ Terminal nội bộ để giao tiếp và nạp file phần mềm `firmware.bin` vào bên trong RAM của thiết bị (hoặc hệ thống mô phỏng).

**Cách giả lập và nạp phần mềm:**

Sau khi biên dịch phần mềm bằng lệnh `make -f Makefile.fw build` để tạo ra `firmware.bin`, bạn khởi chạy phần cứng mô phỏng, và tiến hành nạp mã như sau:

```bash
# Lệnh mở cửa sổ terminal nội bộ của LiteX để đẩy phần mềm vào.
# --kernel firmware.bin: Chỉ định file sẽ nạp vào mạng lưới bộ nhớ phần cứng (RAM).
# localhost:1234       : Khi mô phỏng, mạch ảo lắng nghe tại cổng TCP 1234. 
# (Nếu chạy chip thật, bạn thay bằng đường dẫn thực như /dev/ttyUSB0).

litex_term --kernel firmware.bin localhost:1234
```
*Quá trình này kết nối 2 thành phần độc lập (Mô phỏng Verilator Hardware và Giao diện kết nối Firmware) với nhau.*
